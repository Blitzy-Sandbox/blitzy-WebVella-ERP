The Problem

A single application doing everything — and failing at scale

! Single Point of Failure

One bug in billing crashes the entire ERP — CRM, projects, files, everything goes down together.

↑ Cannot Scale Selectively

Heavy reporting queries slow down data entry for all users. Must scale the entire server to handle one bottleneck.

× Coupled Deployments

A minor CRM fix requires redeploying all 2,256 files. Every release risks breaking unrelated features.

× Security Debt

MD5-hashed passwords, secrets in config files, no per-service access controls. One breach exposes everything.

The current WebVella ERP is a monolithic ASP.NET Core application. All functionality — CRM, invoicing, project management, file storage — runs in a single process with a single PostgreSQL database. A failure in any component takes down the entire system. The app must scale vertically (bigger server) because all code shares one runtime. MD5-hashed passwords represent a critical security vulnerability. Every deployment is all-or-nothing, creating unnecessary risk.

We decomposed the monolith into 11 independently deployable microservices: 10 .NET 9 Lambda services (Native AOT compiled for sub-second cold starts) plus 1 Node.js JWT authorizer Lambda. The frontend is a modern React 19 single-page application served from S3. Each service owns its own datastore — DynamoDB for most services, PostgreSQL for ACID-critical domains (invoicing, reporting), S3 for file storage. Services communicate asynchronously through SNS topics and SQS queues. API Gateway provides a unified entry point with built-in JWT authorization. All infrastructure is defined as code using AWS CDK, deployable to both LocalStack (development) and production AWS with a single codebase.

The biggest cost win is eliminating fixed infrastructure. The monolith requires EC2 instances and RDS databases running 24/7, paying the same rate at 3 AM on Sunday as during peak hours Monday morning. With Lambda and DynamoDB on-demand, we pay per request and per read/write unit. During off-hours when usage drops to near zero, costs drop proportionally. Each service scales independently — if CRM usage spikes, only CRM resources increase. No capacity planning, no over-provisioning, no wasted spend. The 13 CDK infrastructure stacks are fully automated — no manual server management.

• Reliability

No more "the whole ERP is down" incidents

Failure Isolation

If Reporting has an issue, CRM, Invoicing, and Projects continue working normally. Each service fails independently.

Automatic Scaling

Lambda automatically provisions capacity from 0 to thousands of concurrent requests. No capacity planning needed.

Multi-AZ by Default

DynamoDB and Lambda are automatically distributed across multiple data centers. Built-in redundancy at no extra cost.

Dead Letter Queues

Every message queue has automatic retry and failure capture. No lost data, no silent failures.

The monolith's biggest operational risk is blast radius — any failure affects all users and all functionality simultaneously. In the new architecture, each service is isolated. A bug in the reporting engine cannot crash the CRM or invoicing system. Lambda automatically handles scaling and multi-AZ distribution. DynamoDB provides 99.999% availability SLA with global tables. Every SQS queue has a dead letter queue configured — messages that fail processing are captured for investigation, never lost. This architecture eliminates the "3 AM all-hands-on-deck" incidents that occur when the monolith goes down.

✓ Security Posture

From MD5 passwords to enterprise-grade identity management

Cognito-Managed Authentication

AWS Cognito replaces homegrown MD5 password hashing. Industry-standard bcrypt, MFA-ready, and breach-resistant.

Secrets in SSM SecureString

Database credentials and API keys stored encrypted in AWS SSM Parameter Store — never in config files or environment variables.

Least-Privilege IAM

Each service has only the AWS permissions it needs. The CRM Lambda cannot access invoicing data. Defense in depth by design.

Edge Authorization

JWT validation at the API Gateway layer — unauthenticated requests never reach application code. Zero-trust architecture.

The monolith stored passwords using MD5 hashing — a known-broken algorithm that can be cracked in seconds with modern hardware. Cognito provides enterprise-grade identity management: bcrypt password hashing, optional MFA, account lockout policies, and automatic credential rotation. All secrets are stored in AWS SSM Parameter Store with SecureString encryption. The monolith kept database credentials in a plain-text Config.json file. Each Lambda function has a narrowly scoped IAM role — the CRM service cannot read from the invoicing database or S3 bucket. JWT tokens are validated at the API Gateway edge before requests reach any service code, implementing zero-trust principles.

• Risk Mitigation

Multiple safety layers protect the transition

Dual-Target Infrastructure

Same CDK code deploys to LocalStack (development) and AWS (production). Tested locally, confident in production.

5,338 Test Safety Net

Every code change runs through unit, integration, and E2E tests. Regressions are caught before they reach any environment.

Rollback-Safe CDK

CDK deployments are atomic and reversible. A failed deployment automatically rolls back to the previous known-good state.

Gradual Migration Path

Event-driven architecture enables incremental migration — move one service at a time while the others remain stable.

We've built multiple safety layers to de-risk the production transition. The dual-target CDK approach means every infrastructure change is tested locally against LocalStack before touching production AWS. The comprehensive test suite catches regressions at every level — from individual functions to complete user workflows. CDK deployments are transactional: if any resource fails to provision, the entire stack rolls back. No half-deployed states. The event-driven architecture means services are loosely coupled. We can migrate one service at a time — start with low-risk services like file management, then move to critical services like invoicing once confidence is established.
